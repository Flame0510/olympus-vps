# Data Freshness in Olympus

> **Last updated:** 2026-06-30

Understanding how current the data in Olympus is — and what is truly real-time vs. periodically updated.

---

## Session data

**Update frequency:** every 15–30 seconds

The daemon polls `openclaw sessions --json` on this schedule:
- 15 seconds when at least one session has status `working`
- 30 seconds when no sessions are actively working

This means session statuses, token counts, and costs are at most 30 seconds behind reality during quiet periods, and 15 seconds behind during active work.

**What this means for PYTHIA:** if you ask "is Argus working right now?", the answer reflects data that is up to 30 seconds old.

---

## Event feed

**Update frequency:** ~3 seconds (via SSE stream)

The `/api/stream` endpoint pushes updates to the dashboard every ~3 seconds. This includes:
- New events (spawn, complete, error, tool_call)
- Updated session list
- Today's cost total
- Lineage data

The event feed is as close to real-time as Olympus gets. However, events are generated by the daemon's poll cycle, so an event that just happened may take up to 30 seconds to appear.

---

## System metrics (CPU, RAM, disk)

**Update frequency:** every daemon poll cycle (15–30 seconds)

**Retention:** last 24 hours only (older rows are pruned automatically)

---

## Cost totals

**Update frequency:** every 15–30 seconds (recalculated from sessions on each API call)

The cost figures shown in the dashboard are always computed fresh from the database on each page load or SSE push. They are not cached.

**Cost override:** if a manual override is set for the current month, it is shown immediately after being saved — no delay.

---

## Cron jobs

**Update frequency:** on demand (fetched live from OpenClaw gateway on page load)

Cron status (last run, next run) is fetched directly from the OpenClaw gateway each time the Crons page is loaded. It is not stored in the database.

**Note:** if the OpenClaw gateway returns a scope-blocked error when listing crons, Olympus may show no crons even if crons exist. This is an authorization scope issue, not a real absence of crons.

---

## Agent configuration

**Update frequency:** on demand (read from `openclaw.json` on each request)

The agents list, Telegram accounts, and bindings are read directly from `openclaw.json` each time the Agents Config page is accessed. Changes saved via the UI take effect immediately.

---

## Memory / context files

**Update frequency:** on demand (files read from disk on each request)

Memory files (MEMORY.md, etc.) are read from disk each time they are requested. There is no caching.

---

## What is NOT real-time

| Data | Why it may be stale |
|---|---|
| Sessions ended more than 7 days ago | `agents-active` endpoint only looks back 7 days |
| Sessions never polled by daemon | If the daemon was down, sessions from that window are missing |
| Tool calls for a session | Only captured if the OpenClaw session export includes them |
| Costs before Olympus was installed | Historical data before the first daemon run is not available |

---

## How to check if data is fresh

The System Health page shows a check like "last event X seconds ago". If this number is larger than ~60 seconds, the daemon may have stopped.

You can also run this command on the server to check directly:
```bash
sqlite3 /data/.openclaw/workspace-ops/olympus-next-ts/events.db \
  "SELECT datetime(MAX(ts)/1000, 'unixepoch', 'localtime') FROM events"
```
